# Multi-Persona Generator

화이트칼라 작업 영역(이메일, 보고서, PPT, 계약서, 사과문, 이력서, 투자 메모, 사내 공지 등)을 한 단어라도 언급하면, 그 영역 전용 Claude 프로젝트 system instruction을 **멀티 페르소나 방법론**으로 즉시 생성해 주는 Claude Code Skill이다.

생성된 system instruction을 새 Claude 프로젝트의 지침 박스에 그대로 붙여 넣으면, 그 프로젝트에서 만들어지는 모든 산출물이 자동으로 5색 평가 레이어를 통과한다.

> 이 README는 길다. 빠르게 시작하고 싶다면 [60초 시작](#60초-시작) 섹션 하나만 읽으면 된다. 원리·튜닝·내부 동작이 궁금하면 그 아래로 계속 내려간다.

## 60초 시작

1. 스킬 폴더에 클론한다.
   ```bash
   git clone https://github.com/airoasting/multi-persona-generator.git \
     ~/.claude/skills/multi-persona-generator
   ```
2. Claude Code 세션을 새로 열고 한 단어만 친다.
   ```
   이메일
   ```
3. 마크다운 한 묶음으로 system instruction이 출력된다. 끝줄에 *"위 마크다운을 새 클로드 프로젝트의 지침 박스에 붙여 넣어 사용하세요"* 안내가 붙는다.
4. claude.ai에서 새 Project를 만들고, **Project instructions** 박스에 출력된 마크다운을 통째로 붙여 넣는다.
5. 그 프로젝트의 채팅창에 *"고객사에 다음 주 미팅 리마인드 메일 써줘"* 같은 평소 요청을 넣으면, BLACK이 1차 산출물을 만들고 RED·SILVER·BLUE·GOLD 4인이 10점 만점으로 자동 평가한다.

## 무엇을 해결하는가

LLM에 *"이메일 써줘"*라고 던지면 보통 평균치의 글이 나온다. 톤은 무난하고 구조는 표준이며, 어디 한 군데도 틀리지 않았지만 그래서 아무도 다시 읽지 않는다.

이 스킬은 그 평균치를 깬다. 사용자가 작업 영역을 한 마디라도 언급하면, **만드는 사람 1명 + 비판하는 사람 3명 + 읽는 사람 1명**이 등장하는 system instruction을 생성한다. 다섯 명이 모이면 산출물은 만들어지자마자 평가되고, 합격선 미만이면 "v2를 만들까요?"를 묻는다. 자동으로 다시 돌리지 않는다. 다음 행동은 사용자가 결정한다.

핵심 가치 세 가지.

- **명확화 질문 0회.** 사용자가 *"이메일"* 한 단어만 던져도 가장 일반적인 해석으로 즉시 캐스팅한다. *"어떤 이메일 말씀이세요?"*를 묻지 않는다.
- **5색 평가 레이어 자동 장착.** 만드는 즉시 4명이 평가하므로, 사용자가 "다시 평가해줘"를 따로 부르지 않아도 된다.
- **작업 영역별 합격선 차등.** 시·블로그(9.0)와 일반 이메일(9.5)과 사과문·계약서(9.7)의 기준이 다르다. 9.7군은 그 영역의 흔한 실패 문구(예: 사과문의 *"심려를 끼쳐 드린 점 유감"*)를 직접 인용 금지로 박아 넣는다.

## 멘탈 모델: 5색 메타포

| 색 | 역할 | 인원 | 정체 |
|----|------|------|------|
| **BLACK** | 실행자 | 1명 | 그 작업 영역 베테랑 15년차+ |
| **RED** | 비평가 (WHAT 축) | 1명 | 내용·결론·메시지 검증자 |
| **SILVER** | 비평가 (HOW 축) | 1명 | 구조·논리·흐름 검증자 |
| **BLUE** | 비평가 (FORM 축) | 1명 | 톤·분량·시각·호흡 검증자 |
| **GOLD** | 독자 | 1명 | 시간·장소·행위·심리로 정의된 단일 독자 |

다섯 색이 모여야 산출물이 평가된다. 한 명이라도 빠지면 평균치로 돌아간다.

- **BLACK**은 항상 15년차 이상으로 캐스팅한다. *"변호사"*가 아니라 *"M&A 시니어 파트너 15년차+, 면책 누수 직감"*. 톤·제약·분량을 한 줄로 못 박는다.
- **RED·SILVER·BLUE**는 비평가다. 자기 축만 본다. 자기 축이 아닌 영역에 코멘트하면 그 발언은 무효 처리된다.
- **GOLD**는 인구통계가 아니라 상황으로 정의한다. *"30대 직장인"* ✗ → *"월요일 9시 30분, 이사회 직전에 5장짜리 보고서를 처음 펼친 CEO"* ✓. 상황이 사람을 만들기 때문이다.

## WHAT/HOW/FORM 축 독립 원칙

비평가 셋의 시야가 겹치면 평균치로 흐른다. 그래서 RED·SILVER·BLUE 셋의 축은 반드시 다음 세 카테고리에서 **한 개씩** 뽑는다.

| 카테고리 | 무엇을 보는가 | 예시 축 |
|----------|--------------|---------|
| **WHAT** | 내용·결론·메시지·인사이트·핵심 가치 | 의도 전달 / 책임 인정 / 핵심 입장 / 주장의 새로움 |
| **HOW** | 구조·논리·흐름·구성·전개 | 본론·마감·서명 구조 / 역피라미드 / MECE / 후크 위치 |
| **FORM** | 톤·분량·시각·호흡·포맷 | 진정성 톤 / 분량 / 줄바꿈 호흡 / 슬라이드 가독성 |

같은 카테고리에서 두 축을 뽑으면 무효다. 예를 들어 RED와 SILVER 둘 다 *"메시지 명확성"*을 보면 한 명은 잉여다. 반드시 WHAT·HOW·FORM에서 한 명씩.

이 원칙은 [SKILL.md](SKILL.md)의 캐스팅 9원칙 #4번에 박혀 있고, [출력 자체 점검 / 복구 우선순위](#출력-자체-점검--복구-우선순위)에서 위반 시 1순위로 복구된다.

## 설치

### 방법 1. Claude Code skill 폴더에 직접 클론 (권장)

```bash
git clone https://github.com/airoasting/multi-persona-generator.git \
  ~/.claude/skills/multi-persona-generator
```

`~/.claude/skills/` 아래에 폴더가 있으면 Claude Code가 자동으로 인식한다. 별도 등록 절차는 없다. Claude Code를 다시 시작할 필요도 없다. 다음 메시지부터 바로 트리거된다.

### 방법 2. 프로젝트별 설치

특정 프로젝트에서만 사용하고 싶다면 그 프로젝트 루트의 `.claude/skills/` 아래에 클론한다.

```bash
cd /path/to/your/project
git clone https://github.com/airoasting/multi-persona-generator.git \
  .claude/skills/multi-persona-generator
```

이렇게 하면 그 프로젝트 안에서만 트리거되고, 다른 프로젝트에는 영향이 없다. 팀에서 공유할 때도 이 방식이 깔끔하다. 저장소에 커밋해 두면 동료 머신에서도 자동으로 동작한다.

### 방법 3. claude.ai (Claude Code가 아닌 환경)

claude.ai 웹의 Project 환경에서는 skill 자동 로딩이 없다. 대신 [SKILL.md](SKILL.md) 본문(frontmatter 제외)을 통째로 복사해 Project instructions 박스에 붙여 넣으면 같은 동작을 한다. 이 경우 `references/*.md` 파일은 Claude가 읽을 수 없으므로, 자주 쓰는 카테고리(예: external-comm.md) 한두 개도 같은 박스에 이어 붙이면 좋다.

### 동작 확인

설치 후 Claude Code 세션에서 다음을 입력해 본다.

```
이메일 프로젝트 만들어줘
```

스킬이 트리거되면 마크다운 한 묶음으로 system instruction이 출력된다. 끝줄에 *"위 마크다운을 새 클로드 프로젝트의 지침 박스에 붙여 넣어 사용하세요"* 안내가 붙으면 정상 작동.

만약 트리거되지 않거나 평이한 답변이 나온다면 [트러블슈팅](#트러블슈팅) 섹션을 본다.

## 사용법

### 한국어 트리거

작업 영역을 한 마디라도 언급하면 트리거된다. 명확화 질문 없이 즉시 가장 일반적인 해석으로 캐스팅한다.

```
이메일 쓰기
보고서 검토 프로젝트
계약서 검토 지침
사과문 프로젝트
이력서 작업
PPT 지침
SNS 포스트
OKR 리포트
투자 메모
사내 공지
```

단어 하나만 던져도 동작한다.

```
이메일
보고서
PPT
계약서
```

긴 요청도 가능하다. 추가 맥락은 캐스팅에 반영된다.

```
내일 거래처에 보낼 이메일 톤 가이드 좀 잡아줘. 우리 회사 시그니처 톤이 있어야 할 것 같아
사과문 system instruction 필요해. 9.7 보정 적용된 걸로
투자 메모 IC 미팅에서 쓸 거. 가정 명시랑 베이스/업/다운 케이스 강제하는 지침
이력서 이직용. 경력 10년차 IT 매니저
```

### 영문 트리거

```
make a system prompt for resume work
create a project instruction for sales emails
multi-persona evaluation for press release
project guide for OKR reports
system prompt for cover letter
instruction for board memo
```

영문 트리거가 들어오면 자동으로 [`references/english-augment.md`](references/english-augment.md)도 함께 읽어서 영문 산출물 보강을 적용한다. 구체적으로:

- BLACK과 GOLD가 영어 환경 인물로 캐스팅된다 (예: *"Wall Street IR 베테랑 15년차+"*).
- 분량 단위는 단어 수(words)로 바뀐다 (1500~3500자 → 700~1700 words).
- 평가 발화도 영문이 된다.
- *"We apologize if"* 같은 영문 특유의 흔한 실패 문구가 직접 인용 금지에 추가된다.

### 복합 산출물 / 시리즈물

**복합 산출물.** 한 과제가 두 작업 영역에 걸칠 때.

```
영업 덱이랑 발표 스크립트 같이
보고서랑 1페이지 요약
채용 공고랑 사내 공지 동시에
```

이 경우 본체 작업 영역 하나를 결정하고 부속을 BLACK 제약 줄에 흡수한다. 합격선은 둘 중 더 높은 쪽을 따른다(영업 덱 9.5 + 위기 대응 메시지 9.7 → 9.7로 묶는다). 자세한 규칙은 [SKILL.md](SKILL.md)의 *"복합 산출물 라우팅"* 섹션 참고.

**시리즈물.** 한 산출물이 회차로 이어질 때.

```
주간 뉴스레터 시리즈
분기 IR 발표
6주 강의 시리즈
12회 칼럼
```

*"이번 호"·"지난번"·"다음 회"* 같은 단어가 들어오면 자동으로 시리즈로 판단한다. 캐스팅에 *"고정 자산"* 한 줄(매 회차 유지할 것)이 추가되고, 두 번째 회차부터 BLACK 작업 직전에 *"지난 회차 대비 일관성 점검"*이 들어간다.

### 트리거되지 않는 케이스

이미 만든 산출물을 직접 평가·요약·계수하는 작업, 단순 사실 검색, 계산, 기계적 변환은 이 스킬이 담당하지 않는다.

```
이 이메일 톤만 평가해줘
이 보고서 한 줄 요약
이 PPT 슬라이드 수 세줘
이 계약서에 어떤 조항이 있어
이 자소서 점수 매겨줘
summarize this memo in one line
translate this email
fix typos in this report
```

스킬은 *"system instruction을 만드는 도구"*지 *"산출물을 평가·요약하는 도구"*가 아니다. 평가·요약을 원한다면 이 스킬로 만든 프로젝트를 한 번 셋업한 뒤 그 프로젝트 안에서 *"이 이메일 평가해줘"*를 던지면 5색 레이어가 동작한다.

```
    └── trigger_eval.json     트리거/논트리거 테스트 케이스
```

`references/<카테고리>.md` 한 파일 안에는 (1) 매핑 표(작업 영역별 RED·SILVER·BLUE 축, 합격선, 기준 장면), (2) 문체·금지 뱅크 항목, (3) fallback 합성 알고리즘이 들어 있다. 작업 영역을 추가하거나 합격선을 조정하고 싶으면 이 파일들만 손보면 된다. SKILL.md 본문은 카테고리 라우팅 표만 담고 있다.

## 합격선 차등 (9.0 / 9.5 / 9.7)

작업 영역마다 평가 합격선이 다르다. 같은 9점이라도 시(詩)의 9.0과 사과문의 9.7은 의미가 완전히 다르다.

| 합격선 | 작업 영역 성격 | 예 | 특화 금지 개수 | BLACK 제약 |
|--------|---------------|-----|--------------|------------|
| **9.0** | 정답 없음, 개인 톤이 자산 | 시, 단편 소설, 블로그, 자기소개서 일부 | 1~2개 (절제) | 1가지 |
| **9.5** | 표준 (기본값) | 보통의 보고서, 이메일, PPT, 마케팅 카피 | 2~3개 | 1가지 |
| **9.7** | 결함이 곧 손실 | 계약서, 이사회 보고서, 주주 서한, 위기 대응, 사과문, IR 실적 발표 | 4~5개 | 2가지 이상 |

**9.7군의 추가 규칙.** 그 영역의 가장 흔한 실패 문구가 직접 인용으로 최소 두 개 들어간다. 예시:

- 사과문: *"심려를 끼쳐 드린 점 유감"*, *"일부 부주의로"*
- IR: *"견조한"*, *"안정적인 성장세"*
- 계약서: *"합리적 노력"*, *"필요한 경우"*
- 위기 대응: *"~로 보입니다"* (추정 표현), *"일부 부주의로"* (책임 분산)

이렇게 직접 인용으로 박아 넣는 이유는, 일반 어휘로 *"형용사를 줄여라"*라고 적어도 LLM이 그 영역의 진짜 실패 문구를 피하지 못하기 때문이다. 구체적인 문장 단위로 금지를 박아야 그 영역에서 진짜 실수가 줄어든다.

GOLD 합격선 기준 장면도 영역별로 다르다. *"수신자가 미리보기 두 줄로 답장 여부를 정한다"* (이메일), *"기자가 인용할 한 줄을 본문에서 그대로 따올 수 있다"* (사과문 9.7), *"독자가 북마크하거나 링크를 누군가에게 공유한다"* (블로그 9.0). 추상적 점수가 아니라 시간·장소·동작이 들어간 한 줄 장면이다.

## 캐스팅 9원칙

생성되는 system instruction은 다음 9원칙을 따른다.

1. **구체성이 톤을 결정한다.** *"변호사"*가 아니라 *"M&A 시니어 파트너 15년차+, 면책 누수 직감"*으로 좁힌다. 톤은 다섯 가지로 구체화한다. 단어 선택, 문장 호흡, 피하는 것, 시작 패턴, 마무리 패턴.
2. **참조점을 둔다.** *"베인 컨설턴트의 톤"*, *"워런 버핏 주주 서한의 톤"*처럼 비교할 인물이나 사례를 한두 개 적는다.
3. **제약이 페르소나를 굳힌다.** *"무엇을 하지 않는가"*를 적는다. *"추상명사를 쓰지 않는다"*, *"이모티콘을 쓰지 않는다"*.
4. **시야가 겹치면 평균치로 흐른다.** 비평가 셋의 축은 WHAT/HOW/FORM 세 카테고리에서 한 개씩 뽑는다.
5. **상황이 사람을 만든다.** GOLD를 인구통계가 아니라 시간·장소·행위·심리로 정의한다.
6. **왜 신경 쓰는지 적는다.** 변호사라면 *"이 조항으로 클라이언트가 소송에 휘말릴 가능성"*. 슬라이드 디자이너라면 *"회의실 뒷자리에서 글자가 안 보일 가능성"*.
7. **그 분야 언어를 쓴다.** 변호사는 *"조항·리스크·면책·진술과 보증"*. 컨설턴트는 *"MECE·So what·액션 타이틀"*. 그 분야 종사자가 평소 쓰는 단어를 그대로 가져온다.
8. **비평가도 캐스팅한다.** 시각의 출처(15년차+) 한 줄과 발화 톤 한 단어를 함께 적는다. 셋의 발화 톤은 서로 달라야 한다. 풀은 건조·단정·신랄·차분·깐깐·냉정·회의적·무뚝뚝·속내 드러냄·실무자 직설이다.
9. **합격선 기준은 점수가 아니라 장면이다.** *"마지막 한 줄에서만 머뭇거린다"*, *"독자가 한 번만 다시 읽는다"*처럼 시간·장소·동작이 들어간 한 줄.

자세한 내용은 [SKILL.md](SKILL.md)의 *"캐스팅 9원칙"* 섹션 참고.

## 생성되는 system instruction의 구조

생성되는 system instruction은 다음 구조를 따른다(브래킷은 작업 영역에 맞춰 채워진다).

```markdown
# [작업 영역명] 프로젝트

이 프로젝트에서 Claude는 [산출물 종류]의 완성도에 멀티 페르소나 평가 레이어를 더한다.
BLACK이 산출물을 만들면 RED, SILVER, BLUE, GOLD 4인이 10점 만점으로 평가한다.

## 페르소나
- BLACK: [실행자, 15년차+, 톤, 제약, 분량, 합격선]
- RED [축1, WHAT/HOW/FORM 중 한 카테고리, 발화 톤 한 단어]:
    [출처]. 본다: [무엇을]. 어휘. 마무리 패턴. (합격선)
- SILVER [축2, RED와 다른 카테고리, 발화 톤]: ...
- BLUE [축3, RED·SILVER와 다른 카테고리, 발화 톤]: ...
- GOLD [상황 한 줄, 시간·장소·행위·심리]:
    어휘. 마무리 패턴. (합격선)

## 워크플로우
1. BLACK이 1차 산출물을 만든다.
2. RED, SILVER, BLUE, GOLD 순으로 평가한다.
3. 분석 3인 평균과 GOLD 체류도를 따로 보고한다.
4. 합격선 미달이면 "v2를 만들까요?" 묻는다.

## 점수 기준
[10/9.5/9/8/6-7/4-5/3 기준 장면]
[작업 영역별 합격선 보정: 9.0/9.5/9.7]

## 문체 지침
공통 (모든 작업 영역) + [작업 영역] 특화

## 하지 말아야 할 것
공통 (정량 임계값) + [작업 영역] 특화 금지

## 적용하지 않는 경우
[빠른 작업이 명시된 경우 BLACK만 단독 실행]
```

각 섹션의 정확한 형식은 [SKILL.md](SKILL.md)의 *"출력 템플릿"* 섹션에 박혀 있다. 풀 출력 예시 5종은 [`references/examples.md`](references/examples.md)에 들어 있다.

## 생성된 프로젝트 안에서 일어나는 일 (워크플로우)

생성된 system instruction을 새 Claude 프로젝트의 지침 박스에 붙여 넣은 뒤, 그 프로젝트의 채팅창에 평소 요청을 던지면 다음 순서로 동작한다.

1. **BLACK이 1차 산출물을 만든다.** 캐스팅된 톤·제약·참조점·분량을 따라간다. 자평이나 *"어떠신가요"* 같은 마무리 문구는 붙이지 않는다.
2. **RED → SILVER → BLUE → GOLD 순으로 평가한다.** 각 발화 직전에 색깔 라벨을 다시 적는다. 점수는 그 페르소나의 합격선 기준 장면과 비교해 매긴다.
3. **분석 3인 평균과 GOLD 체류도를 따로 보고한다.** 두 값은 합쳐지지 않는다. 분석 3인은 "기준에 닿았는가"를 보고, GOLD는 "끝까지 머물렀는가"를 본다. 두 측정은 독립적이어야 한다.
4. **합격 판정.** 분석 3인 모두 합격선 이상이고 GOLD 체류도도 합격선 이상이면 합격이다. 어느 하나라도 합격선에서 0.5점 이상 낮으면 *"v2를 만들까요?"*라고 사용자에게 묻는다. **자동으로 다시 돌리지 않는다.** 다음 행동은 사용자가 결정한다.
5. **수정 라운드.** v2를 진행하면 GOLD 체류도가 가장 낮은 축의 수정안을 먼저 반영한다. 권장 최대는 4라운드. 같은 축이 라운드 2 이후에도 합격선에서 0.5점 이상 낮으면 캐스팅 또는 과제 정의 문제일 가능성을 사용자에게 알린다.

이 흐름이 매번 일관되게 동작하는 것이 이 스킬의 핵심 가치다. 사용자가 *"평가해줘"*나 *"다시 써줘"*를 따로 부르지 않아도 자동으로 5색 레이어가 들어간다.

## 출력 분량 가이드

생성하는 system instruction은 한 마크다운 블록 안에서 1500~3500자(영문 700~1700단어)다.

| 작업 영역 성격 | 분량 |
|---------------|------|
| 단순 영역 | 1500~2000자 |
| 표준 (대부분) | 2000~2800자 |
| 9.7군 또는 복합 산출물 | 2800~3500자 |

1500자 미만이면 페르소나·문체·금지의 디테일이 부족해 평가 레이어가 작동하지 않는다. 3500자 초과면 사용자가 클로드 프로젝트 지침 박스에 붙일 때 압도된다.

## 출력 자체 점검 / 복구 우선순위

스킬은 출력 직전에 자체 점검을 통과시킨다. 위반 시 다음 우선순위로 복구한다.

| 우선순위 | 위반 영역 | 복구 동작 |
|----------|-----------|-----------|
| 1 | 축 독립성 (같은 카테고리 두 축) | 비평가 한 명을 다른 카테고리로 재캐스팅 |
| 2 | 합격선 강도 미달 (9.7군 금지 부족) | 흔한 실패 문구 두 개를 직접 인용으로 추가, BLACK 제약도 한 줄 추가 |
| 3 | GOLD가 인구통계 (상황 부재) | 시간·장소·행위·심리 네 요소를 채워 한 줄 장면으로 다시 적기 |
| 4 | 합격선이 추상 (*"여지가 남는다"*) | 시간·장소·동작이 들어간 한 줄 장면으로 교체 |
| 5 | 발화 톤 중복 | 풀에서 다른 단어로 교체 |
| 6 | 분량 가이드 누락 | 참조 파일의 그 작업 영역 분량 항목으로 채우기 |
| 7 | 끝줄 안내 문구 누락 | *"위 마크다운을 새 클로드 프로젝트의 지침 박스에 붙여 넣어 사용하세요"* 추가 |
| 8 | 자평·메타 코멘트 (블록 안에 *"어떠신가요"*) | 그 줄 삭제 |

위 항목을 고치면 아래 항목이 따라 풀리는 경우가 많다. 그래서 위에서부터 고친다. 자세한 점검 항목 전체는 [SKILL.md](SKILL.md)의 *"출력 시 자체 점검"* 섹션 참고.

## 평가 (Evals)

[`evals/`](evals/) 디렉터리에 트리거 정확도 평가 셋이 있다. 트리거되어야 하는 발화와 트리거되면 안 되는 발화를 모은 JSON으로, 스킬을 수정한 뒤 회귀 테스트용으로 사용한다.

### 트리거링 evals

[`evals/trigger_eval.json`](evals/trigger_eval.json)에는 should-trigger 10개와 should-not-trigger 10개가 들어 있다. should-not-trigger 10개 중 8개는 **near-miss**다. *"이메일·보고서·PPT·이력서·자소서·계약서"* 같은 키워드를 포함하지만 사용자가 만든 산출물을 평가·요약·계수하는 작업이라서 트리거되면 안 된다. 마지막 2개만 완전 무관(점심 메뉴·코딩)이다.

skill-creator의 `run_loop.py`로 description을 자동으로 5번 반복 개선하면서 트리거 정확도를 측정할 수 있다. 60% 학습 + 40% 테스트 분할로 오버피팅을 방지한다. 합격선은 테스트 점수 0.85 이상.

### 출력 품질 evals

`run_eval.py`는 자동 grader 기반인데 이 스킬은 출력이 창의적 system instruction이라 자동 grading이 어렵다. 수동 평가가 더 정확하다. 다음 5개 케이스를 각각 3회 호출해 일관성을 본다.

| ID | 입력 | 검증 항목 |
|----|------|-----------|
| Q1 | *"이메일"* | 9.5군. 카테고리 라우팅 정확. 분량 1500~2800자 |
| Q2 | *"사과문"* | 9.7군. 직접 인용 금지 2개 이상. BLACK 제약 2개 |
| Q3 | *"블로그"* | 9.0군. 특화 금지 1~2개로 절제 |
| Q4 | *"영업 덱이랑 발표 스크립트 같이"* | 복합 산출물. 본체+부속 흡수. GOLD 단일 독자 |
| Q5 | *"make a system prompt for board memo"* | 영문 트리거. english-augment 적용. GOLD 영문 환경 |

각 출력별로 9개 체크리스트를 통과해야 한다. 자세한 실행 방법과 합격 기준은 [`evals/EVALS_GUIDE.md`](evals/EVALS_GUIDE.md) 참고.

### 60초 sanity check

evals 다 돌릴 시간이 없을 때.

1. 새 클로드 프로젝트를 만들고 SKILL.md 본문을 지침 박스에 붙인다.
2. 채팅에 *"이메일"*이라고만 친다.
3. 응답이 마크다운 한 묶음으로 오고 9개 체크리스트가 모두 통과하는지 확인한다.
4. *"사과문"* 한 번 더 쳐서 9.7군 처리(직접 인용 금지 2줄 이상)가 들어가는지 확인한다.

이 둘이 통과하면 일상 사용 OK 수준.

## 자주 묻는 질문 (FAQ)

**Q. 왜 명확화 질문을 안 묻나? 사용자 의도를 정확히 파악해야 하지 않나?**

명확화 질문은 비용이다. *"이메일"* 한 단어를 던진 사용자에게 *"어떤 이메일이요? 사외? 사내? 톤은?"*을 묻는 순간, 사용자는 자기가 모호하게 던진 것을 후회하고 화면을 닫는다. 이 스킬은 가장 일반적인 해석으로 즉시 캐스팅하고, 사용자가 다른 변형을 원하면 다음 메시지에서 직접 말하게 한다. 한 번에 끝나는 경우가 70% 이상이고, 안 맞으면 다음 메시지가 더 빠르다.

**Q. 5색 평가 레이어가 매번 동작하면 답변이 너무 느려지지 않나?**

느려진다. 그게 이 스킬의 비용이다. 5색이 모이는 시점에서 평가 합격선을 하나라도 못 넘으면 사용자에게 *"v2를 만들까요?"*를 묻고 멈춘다. 자동으로 다시 돌지 않는다. 그래서 시간을 통제할 수 있다. 빠른 작업이 필요하면 *"그냥 빨리 써줘"*를 명시한다. 그러면 BLACK만 단독으로 실행된다.

**Q. 같은 입력에 대해 매번 다른 사람이 캐스팅된다. 일관성이 없는 것 같다.**

이 스킬의 일관성은 *"매번 같은 사람"*이 아니라 *"매번 같은 범위 안의 사람"*이다. *"이메일"* 입력에 대해 BLACK이 한 번은 *"15년차+ 외부 영업 디렉터"*, 다음에는 *"15년차+ 비즈니스 PM"*으로 바뀔 수 있다. 둘 다 *"외부 비즈니스 이메일 베테랑 15년차+"* 범위 안이라 일관성 통과다. 만약 한 번은 변호사, 다음은 광고 카피라이터로 바뀐다면 그건 진짜 흔들림이다. 이런 흔들림은 [출력 품질 evals](evals/EVALS_GUIDE.md)의 일관성 체크가 잡아낸다.

**Q. 작업 영역이 카테고리에 없으면?**

가장 가까운 카테고리로 라우팅된다. 카테고리도 모호하면 외부 커뮤니케이션이 최종 fallback이다. 새 작업 영역을 추가하고 싶으면 해당 카테고리의 references 파일에 매핑 행 한 줄과 문체·금지 뱅크 항목을 추가하면 된다. SKILL.md 본문은 손댈 필요가 없다.

**Q. claude.ai에서 쓸 수 있나?**

쓸 수 있지만 자동 로딩은 안 된다. SKILL.md 본문(frontmatter 제외)을 통째로 복사해 Project instructions 박스에 붙여 넣으면 동일 동작을 한다. 단 references 파일은 못 읽으므로 자주 쓰는 카테고리 한두 개를 같이 붙이면 좋다.

**Q. 영문과 한국어를 섞어 쓰면?**

트리거 단어가 섞여 있으면 한국어 우선으로 캐스팅한다. *"create a project for 사과문"*은 한국어 트리거(*"사과문"*)로 처리되어 사과문 9.7군 한국어 출력이 나온다. 명확히 영문 산출물을 만들고 싶으면 *"make a system prompt for [작업 영역] in English"* 형태로 적는다.

**Q. 이미 만든 프로젝트를 평가만 받고 싶은데.**

이 스킬은 평가만 하는 도구가 아니다. 이 스킬로 *"[작업 영역] 평가 프로젝트"*를 한 번 만든 뒤(예: *"이메일 평가 시스템 만들어줘"*), 그 프로젝트 안에 평가받을 산출물을 던지면 5색 레이어가 동작한다. 또는 자체 평가만 원한다면 BLACK 없이 RED·SILVER·BLUE·GOLD 4인 평가만 돌리는 별도 system instruction을 직접 작성하는 편이 더 적합하다. 그 작업은 이 스킬의 범위를 벗어난다.

## 트러블슈팅

**스킬이 트리거되지 않는다.**

- `~/.claude/skills/multi-persona-generator/` 경로에 폴더가 있는지 확인. SKILL.md 파일이 그 안에 있어야 한다.
- 폴더 이름이 정확한지 본다. `multi-persona-generator` (영문 소문자, 하이픈)이어야 한다.
- Claude Code 세션을 새로 열어 본다. 이미 열린 세션은 새 스킬을 인식하지 못할 수 있다.
- *"코드 작성해줘"* 같은 비-화이트칼라 작업 트리거는 의도적으로 잡지 않는다. *"이메일"·"보고서"·"PPT"* 같은 작업 영역 단어를 명시한다.

**트리거되긴 하는데 평이한 답변이 나온다.**

- references 디렉터리의 카테고리 파일이 모두 있는지 확인. SKILL.md만 있고 references가 없으면 매핑 행을 못 읽어 페르소나 디테일이 빠진다.
- 출력 끝줄에 *"위 마크다운을 새 클로드 프로젝트의 지침 박스에 붙여 넣어 사용하세요"* 안내가 있는지 확인. 이 안내가 빠지면 출력 절차 단계가 한두 개 누락됐을 가능성이 있다.

**9.7군인데 직접 인용 금지가 1개뿐이거나 없다.**

- 자체 점검 [복구 우선순위 #2](#출력-자체-점검--복구-우선순위)에 해당. 사용자가 다음 메시지에 *"9.7군이면 직접 인용 금지 두 개 이상이어야 함"*을 명시하면 재생성된다.
- 또는 evals를 돌려 description의 9.7 분기 트리거 정확도를 측정하고, 필요시 description을 갱신한다.

**복합 산출물인데 두 영역이 따로 캐스팅됐다.**

- 본체-부속 흡수가 안 된 케이스. 사용자가 *"본체는 [영역A], 부속으로 [영역B] 흡수"*를 명시한다.
- 두 영역의 합격선이 다르면 더 높은 쪽이 자동으로 적용된다. 그게 안 되면 캐스팅 9원칙 #4 위반(축 독립성)일 가능성이 크다. 비평가 셋이 서로 다른 카테고리에서 뽑혔는지 확인한다.

**영문 트리거인데 한국어 출력이 나온다.**

- 트리거 단어가 한국어와 영문이 섞여 있을 가능성. 명확히 *"in English"* 또는 *"system prompt in English for [작업 영역]"* 형태로 다시 시도한다.
- 영문 출력이 일관되게 안 나오면 [`references/english-augment.md`](references/english-augment.md) 파일이 폴더에 있는지 확인.

## 기여하기

이슈와 PR을 환영한다.

- **새 작업 영역 추가.** 해당 카테고리의 references 파일에 매핑 행 한 줄과 문체·금지 뱅크 항목을 추가하는 PR을 보낸다. SKILL.md 본문은 카테고리 라우팅 표만 담고 있으므로 새 카테고리가 아니라면 손댈 필요 없다.
- **새 카테고리 추가.** 9번째 카테고리가 필요하다고 판단되면 (1) 새 references 파일 작성, (2) SKILL.md의 1차 라우팅 표에 행 추가, (3) 트리거 evals에 should-trigger 케이스 2~3개 추가 PR.
- **Evals 케이스 추가.** 트리거가 잘못되는 케이스를 발견하면 [`evals/trigger_eval.json`](evals/trigger_eval.json)에 케이스를 추가해 PR을 보낸다.
- **합격선 조정.** 어느 영역이 9.5인데 9.7이어야 한다고 판단되면 issue로 근거(흔한 실패 사례)와 함께 제안한다.

PR 전에 다음을 확인한다.

- references 파일의 매핑 표 형식(헤더 4열, 과제 / RED / SILVER / BLUE / 합격선)을 따랐는가.
- 새 작업 영역의 합격선 기준 장면이 시간·장소·동작이 들어간 한 줄인가.
- 9.7군이라면 직접 인용 금지 2개 이상이 들어갔는가.
- 60초 sanity check를 통과하는가.
